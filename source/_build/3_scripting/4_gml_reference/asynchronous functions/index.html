Funciones asincronas


Antes de tratar con las funciones asíncronas (referidas de ahora en adelante como funciones asíncronas ) en GameMaker Studio 2 es posible que se esté preguntando qué significa realmente la palabra "asíncrono"! Bueno, la definición real del diccionario de asíncrono es:

"perteneciente a una técnica de transmisión que no requiere un reloj común entre los dispositivos de comunicación; las señales de temporización se derivan de caracteres especiales en el flujo de datos en sí".

En el contexto de GameMaker Studio 2, una función asíncrona es una que se envía a un servidor web, o solicita la entrada del usuario, o incluso transmite datos, mientras que permite que GameMaker Studio 2 continúe ejecutándose sin bloqueo. Cuando se recibe la información de devolución de llamada , se dispara un evento asíncrono especial que, a su vez, permite que cualquier instancia con un evento asíncrono asignado a él ejecute funciones adicionales (aunque esto no es así). Entonces, básicamente, es una forma de comunicarse con algún recurso externo (servidor, usuario o dispositivo) sin que el juego o el proyecto se bloquee mientras se espera una respuesta.

¿Porque es esto importante? Bueno, ya que permite que GameMaker Studio 2 continúe funcionando mientras se envían o reciben datos, y esto significa que puedes hacer muchas cosas sin bloquear el bucle del juego como:

  • transmite datos del dispositivo a su juego sin que el jugador esté esperando a que se carguen las cosas
  • tiene eventos de devolución de llamada para hacer cosas solo cuando se recibe la información correcta
  • Comunicar e intercambiar datos con un servidor web.

Todo esto hace que todo funcione de una manera mucho más fluida y discreta. Debe tenerse en cuenta que los eventos asíncronos se activarán para todas las instancias que los tengan, por lo que puede usar una función asíncrona en una instancia y tener el evento asíncrono que trata la respuesta en otra, o incluso en otras. Puede ir aquí para obtener más información sobre cómo funciona el evento asíncrono: Eventos asíncronos.

Tenga en cuenta que las funciones http_ anteriores pueden no funcionar como se espera debido a problemas de seguridad entre dominios. Esto significa que las solicitudes a su servidor o los intentos de cargar recursos de dominios están bloqueados y puede parecer que devuelven resultados en blanco o errores 404. Una de las formas en que puede solucionar esto es tener algún PHP del lado del servidor que permita que ciertos dominios accedan a su servidor (esta también es una forma de proteger sus recursos y bloquear servidores que no están incluidos en la lista de permisos de PHP).

El siguiente es un ejemplo del código que puede usar para esto:

$http_origin = $_SERVER['HTTP_ORIGIN'];
if ($http_origin == "http://127.0.0.1:51268")
   {
   header('Access-Control-Allow-Origin: *');
   }

Para las solicitudes de carga de imágenes donde es importante determinar o configurar su tipo de origen cruzado, existen las siguientes funciones:

  • http_get_request_crossorigin
  • http_set_request_crossorigin

En la mayoría de los casos, estas funciones no deberían ser utilizadas, pero si el juego se almacena en un servidor seguro, es posible que ciertos recursos requieran el acceso a la autenticación básica y generen errores de seguridad al cargar: establecer el tipo de origen cruzado para las solicitudes de imágenes para "utilizar credenciales" puede ser necesario (a diferencia de la configuración predeterminada "anónima").


Las versiones anteriores de GameMaker tenían diálogos emergentes que bloqueaban el corredor mientras esperaban una respuesta del usuario. Sin embargo, a la mayoría de los dispositivos modernos no les gusta esto e interpretarán el paro del corredor como un error y cerrarán la aplicación. Para evitar que este GameMaker Studio 2 tenga equivalentes asíncronos de estas funciones que cumplen el mismo propósito que las versiones anteriores, pero no bloquean el corredor del juego en ningún momento, lo que significa que, cuando se llame, el juego se ejecutará normalmente en segundo plano mientras el usuario responde. El diálogo mostrado. Una vez que el usuario haya respondido, se llamará a un evento asíncrono de diálogos para que pueda manejar los resultados.

Al hacer juegos, a menudo es necesario almacenar información sobre el estado del juego en un archivo de algún tipo, pero almacenar esta información en el dispositivo no siempre es la mejor opción ya que, si el jugador eliminó el juego y luego lo vuelve a instalar, este La información puede perderse. Para evitar esto, puede utilizar varios servicios en la nube diferentes, que ofrecen almacenamiento de datos a través de Internet para su recuperación y modificación en cualquier momento. Debe tener en cuenta que esta función está limitada a un solo "blob" de datos por juego, por lo que cada vez que envíe un nuevo guardado al servicio en la nube, ya sea una cadena o un archivo, sobrescribirá los datos almacenados anteriormente.

GameMaker Studio 2 admite esto con algunas funciones simples que funcionan en conjunto con el Evento Asincrónico en la Nube. Estas funciones se explican en las siguientes páginas:

  • cloud_synchronise
  • cloud_string_save
  • cloud_file_save
NOTA: Actualmente, solo el objetivo de Android utiliza estas funciones, y debe marcar la casilla de verificación Habilitar Google Cloud Saving en la sección Social de Android Opciones de juego de Android. Esto le pedirá que descargue la extensión de Servicios de Google, que es necesaria para que funcione el ahorro en la nube en Android. En los dispositivos móviles, normalmente solo una aplicación puede estar activa en primer plano en cualquier momento, pero muchos juegos y aplicaciones funcionan en un entorno interconectado o basado en el tiempo donde pueden ocurrir eventos de interés para los usuarios cuando la aplicación no está en primer plano. En estos casos, las notificaciones push locales y remotas pueden permitir que estos juegos notifiquen a sus usuarios cuando ocurren eventos.

Cuando el sistema operativo entrega una notificación de inserción local o remota y la aplicación de destino no se ejecuta en primer plano, presenta la notificación al usuario en forma de un banner. Si hay una alerta de notificación y el usuario la pulsa, entonces se inicia el juego y se llama un Evento de Empuje Asíncrono donde puede verificar la cadena de devolución de llamada que especifique y resolver el empuje según su valor. Si la aplicación se ejecuta en primer plano cuando se entrega la notificación, la aplicación seguirá recibiendo un evento de inserción asíncrona, pero el usuario no mostrará ningún mensaje a menos que lo cree usted mismo.

Hay algunas diferencias a tener en cuenta entre las plataformas compatibles:

  • En Android, todas las notificaciones recibidas cuando la aplicación no se ejecuta / en segundo plano, se almacenarán, y los datos entregados por el evento asíncrono la próxima vez que se ejecute la aplicación, y deberá habilitar las notificaciones automáticas dentro de Android Opciones de juego de Android para que trabajo.

  • En iOS, los datos para una notificación en particular solo se entregarán cuando se seleccione la notificación (o si la aplicación se está ejecutando en primer plano cuando se recibe la notificación) y deberá habilitar las notificaciones automáticas dentro de las Opciones de juego de iOS para que funcionen.

Una notificación de inserción local es local al dispositivo en el que está instalado el juego y no requiere un servidor backend. Simplemente establece la fecha y la hora para la notificación y se mostrará al usuario si el juego no se está ejecutando actualmente. Este tipo de notificación es útil para configurar "recordatorios" para que el usuario juegue su juego nuevamente, u ofrezca una recompensa diaria por jugar, etc... y cuando el usuario toca la notificación, lanzará el juego y pasará una información. cadena que luego puede ser analizada por el juego para dar una recompensa o lo que sea.

Las siguientes funciones están disponibles para notificaciones locales:

  • push_local_notification
  • push_get_first_local_notification
  • push_get_next_local_notification
  • push_cancel_local_notification
NOTA: Esta funcionalidad está limitada a los módulos de destino de iOS y Android. Para Android, deberá haber instalado la extensión de servicios de Google Play y haber habilitado las notificaciones push en las Opciones de juegos sociales de Android.

Los mensajes de notificación remota son enviados por un servidor a un servicio provisto por la plataforma App Store de dispositivos, y esto luego reenvía esos mensajes a todos los dispositivos en los que está instalada su aplicación. Esto es compatible con GameMaker Studio 2 en los módulos de destino de iOS y Android.

No hay funciones en GameMaker Studio 2 para lidiar con las notificaciones remotas, ya que todas deben ser generadas por su servidor y manejadas por las respectivas tiendas de aplicaciones. Sin embargo, una vez configurados correctamente, los juegos de GameMaker Studio 2 recibirán estas notificaciones, que luego pueden tratarse en el evento de empuje asíncrono, como lo haría una notificación local.

Para todas las plataformas disponibles, una vez que haya realizado la configuración necesaria, cuando el juego se ejecute en un dispositivo, registrará ese dispositivo con el servicio de notificación de inserción de plataformas. Esto activará un evento de notificación push asíncrono, y el ds_map la clave " tipo " tendrá el valor " registro " así como una nueva clave, " reg_id ", que contiene el ID de registro (o un mensaje de error en la clave " error ", si " estado " es 0). Luego, debe enviar este ID de registro a su servidor, y cada dispositivo en el que esté instalado su juego tendrá un ID de registro diferente. Su servidor debe mantener una lista de identificadores para los dispositivos registrados, como cuando, cuando envía un mensaje de notificación de inserción desde su servidor, utiliza los identificadores de registro para enviar el mensaje a los dispositivos registrados.

Tenga en cuenta que no hay garantía de que se enviarán notificaciones push remotas, y que la carga útil de datos permitida es bastante pequeña. Esto varía según las plataformas, pero iOS es particularmente limitado: el servicio de Apple solo entrega la notificación más reciente, que debe ser seleccionada por el destinatario para que los datos de carga útil se entreguen a su evento asíncrono, y estas notificaciones tienen un tamaño máximo de carga útil de 256 bytes.. Por lo general, una notificación de inserción remota simplemente indicaría que hay nuevos datos disponibles desde su servidor, por ejemplo.

NOTA: Android requiere que agregue el ID de remitente de GCM en Android Opciones de juego de Android. Este es el Número de Proyecto que se asigna cuando crea su Proyecto API de Google Play.
