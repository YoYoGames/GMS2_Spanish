Comprobación de colisiones


Al planificar movimientos o decidir ciertas acciones, a menudo es importante ver si hay colisiones con otros objetos en ciertos lugares dentro del mundo del juego y, a menudo, elegir la colisión correcta para el trabajo es la tarea más importante de todas. GameMaker Studio 2 tiene una serie de funciones integradas para ayudarte a enfrentar las colisiones correctamente y de acuerdo con las necesidades de tu proyecto.

¡IMPORTANTE! Todas las funciones de colisión funcionan con posiciones enteras, independientemente de los valores de entrada. Cualquier valor no entero se redondeará al entero más cercano.


Las siguientes funciones se ocupan de las diversas formas en que puede hacer que una instancia verifique las colisiones con otra instancia:

  • place_empty
  • place_free
  • place_meeting
  • position_empty
  • position_meeting
  • cambio de posición
  • position_destroy
  • lugar_instancia
  • instance_place_list
  • instance_position
  • instance_position_list

Las siguientes rutinas se pueden usar para esto (así como otras funciones detalladas en las secciones relacionadas con Desplazamientos e instancias ). Todas estas funciones de colisión funcionan de manera ligeramente diferente, pero mantienen tres argumentos comunes que explicaremos aquí:

  • obj Aquí es donde especificamos el objeto con el que la función debe verificar una colisión. Puede ser un ID de instancia, un tipo de objeto, o la palabra clave GM especial todos. También vale la pena señalar que si el tipo de objeto elegido es el padre designado de otros objetos, entonces todos los hijos de ese objeto también se incluirán en esta verificación.


  • prec. Esto puede ser cualquiera true o false y se relaciona con la precisión de la comprobación de colisiones. Si se establece en true, luego se deben calcular las colisiones precisas (pero solo si el objeto que se está verificando tiene marcado "preciso" para su detección de colisiones en sus propiedades de máscara), que es una tarea un poco más compleja para la CPU y, por lo tanto, se debe usar con precaución. Si false luego, las colisiones se calcularán en función del cuadro delimitador del objeto que se va a verificar.


  • no me Es posible que deba verificar las colisiones con un objeto que tenga el mismo índice que el objeto que ejecuta el código, por lo que para evitar que el código encuentre una colisión consigo mismo, puede configurar esto true. Si está buscando específicamente la instancia del objeto que ejecuta el código, o no es importante, se puede dejar como false.

Las funciones básicas devolverán el ID de una instancia encontrada como colisionable o la palabra clave especial noone cuando no hay colisión, mientras que las funciones de la lista devolverán el número de instancias que están en colisión y rellenarán una lista de DS prefabricada con sus ID. Tenga en cuenta que si hay varias colisiones con las áreas definidas por estas funciones e instancias del objeto dado, solo se devuelve un ID de instancia, y puede ser cualquiera de las instancias de la colisión.

Existen las siguientes funciones que se ocupan de las colisiones avanzadas.

  • colision_circle
  • collision_circle_list
  • collision_ellipse
  • collision_ellipse_list
  • collision_line
  • collision_line_list
  • punto de colisión
  • collision_point_list
  • colision_rectangle
  • collision_rectangle_list

Todas las funciones anteriores están relacionadas con las instancias de comprobación de colisiones y, como tales, dependen de la máscara de colisión definida para la instancia. Sin embargo, hay muchos momentos en los que es necesario verificar las "colisiones" con un punto o un área, especialmente cuando su instancia no tiene un sprite asignado, o cuando está trabajando con el mouse, etc... Por lo tanto, GameMaker Studio 2 también proporciona las siguientes funciones para ayudarle en estas situaciones:

  • point_in_rectangle
  • punto_en_triángulo
  • point_in_circle
  • rectangle_in_rectangle
  • rectangle_in_triangle
  • rectangle_in_circle

Al utilizar la compilación en física, no se garantiza que las funciones de colisión anteriores funcionen para las instancias habilitadas para física. Esto se debe principalmente al hecho de que estas instancias ya no usan la mayoría de las variables integradas regulares (en su lugar, las instancias habilitadas por la física tienen su propio conjunto de variables ) y tampoco usan la máscara de colisión o el cuadro delimitador, ya que utilizan dispositivos fijos en lugar. Sin embargo, hay una función especial para probar las colisiones con la física habilitada que comprueba los dispositivos superpuestos:

  • physics_test_overlap
