Verificación de colisión

Al planear movimientos o decidir sobre ciertas acciones, a menudo es importante ver si hay colisiones con otros objetos en ciertos lugares dentro del mundo del juego, y a menudo elegir la colisión correcta para el trabajo es la tarea más importante de todas. GameMaker Studio 2 tiene una serie de funciones integradas para ayudarlo a manejar las colisiones correctamente y de acuerdo con las necesidades de su proyecto.

¡IMPORTANTE! Todas las funciones de colisión funcionan usando posiciones enteras, independientemente de los valores de entrada. Cualquier valor no entero se redondeará al entero más cercano.


Las siguientes funciones se refieren a las diversas formas en que puede tener una instancia de verificación de colisiones con otra instancia:

  • place_empty
  • place_free
  • place_meeting
  • position_empty
  • position_meeting
  • position_change
  • position_destroy
  • instance_place
  • instance_position

Las siguientes rutinas pueden usarse para esto (así como para otras funciones detalladas en las secciones relacionadas con Moving Around and Instances ). Estas funciones de colisión funcionan de manera ligeramente diferente, pero mantienen tres argumentos comunes que explicaremos aquí:

  • Obj. Aquí es donde especificamos el objeto que la función debe verificar para detectar una colisión. Puede ser un ID de instancia, un tipo de objeto, o la palabra clave GM especial todos. También vale la pena señalar que si el tipo de objeto elegido es el padre designado de otros objetos, todos los hijos de ese objeto también se incluirán en este control.


  • prec. Esto puede ser cualquiera true o false y se relaciona con la precisión de la verificación de colisión. Si está configurado para true, entonces se deben calcular las colisiones precisas (pero solo si el objeto que se está comprobando tiene marcada "precisa" para su detección de colisión en sus propiedades de máscara), que es una tarea ligeramente más compleja para la CPU, por lo que debe usarse con precaución. Si false entonces las colisiones se calcularán basándose en el cuadro delimitador del objeto a verificar.


  • notme. Es posible que deba verificar las colisiones con un objeto que tenga el mismo índice que el objeto que ejecuta el código, por lo que para evitar que el código encuentre una colisión consigo mismo, puede configurarlo para true. Si está buscando específicamente la instancia del objeto que ejecuta el código, o no es importante, puede dejarlo como false.

Todas estas funciones devuelven el id de una instancia que se encuentra en colisión o la palabra clave especial noone cuando no hay colisión. Tenga en cuenta que si hay múltiples colisiones con las áreas definidas por estas funciones e instancias del objeto dado, solo se devuelve una ID de instancia, y puede ser cualquiera de las instancias en la colisión.

Las siguientes funciones existen que se ocupan de colisiones avanzadas.

  • collision_circle
  • collision_ellipse
  • collision_line
  • collision_point
  • collision_rectangle

Todas las funciones anteriores están relacionadas con las instancias de comprobación de colisiones y, como tales, se basan en la máscara de colisión definida para la instancia. Sin embargo, hay muchos momentos en los que necesita verificar "colisiones" con un punto o un área, especialmente cuando su instancia no tiene un sprite asignado, o cuando está trabajando con el mouse, etc.... Por lo tanto, GameMaker Studio 2 también proporciona las siguientes funciones para ayudarlo en estas situaciones:

  • point_in_rectangle
  • point_in_triangle
  • point_in_circle
  • rectangle_in_rectangle
  • rectangle_in_triangle
  • rectangle_in_circle

Al utilizar la compilación en física, no se garantiza que las funciones de colisión anteriores funcionen para las instancias habilitadas para la física. Esto se debe principalmente al hecho de que estas instancias ya no usan la mayoría de las variables regulares incorporadas (en cambio, las instancias habilitadas con física tienen su propio conjunto de variables ) y tampoco usan la máscara de colisión o el cuadro delimitador, ya que utilizan accesorios en lugar. Sin embargo, hay una función especial para probar colisiones con física habilitada que verifica la superposición de dispositivos:

  • physics_test_overlap
