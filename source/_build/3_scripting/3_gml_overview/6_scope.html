Variables y alcance variable


Como cualquier lenguaje de programación, GML utiliza variables como la unidad básica para la mayoría de las operaciones de programación. Las variables se utilizan para almacenar información en la memoria del dispositivo para su uso posterior (o instantáneo), y se les da un nombre para que pueda consultarlas en funciones y scripts. Una variable en GML puede almacenar muchos tipos de datos diferentes, como un número real (como 100, 2.456575, -56, etc.), una cadena (como "¡Hola mundo!"), Un número entero (como 1, 556, -7 ), o un booleano ( true o false ).

Una variable es algo que usamos para almacenar un valor para usar en una o más operaciones o funciones. Piense en "pi", por ejemplo... es una variable del mundo real que contiene el valor 3.14159265 (etc...). ¿Por qué? Bueno, ¡es mucho más fácil decirle a alguien "pi" que "tres puntos uno cuatro uno cinco nueve dos seis cinco"! Entonces, nombrar cosas hace que la vida sea mucho más sencilla y también significa que si el valor de esa variable cambiara, no tenemos que cambiar el número en todas partes, ya que el nombre de la variable sigue siendo el mismo. En GML, una variable tiene un nombre que debe comenzar con una letra y puede contener solo letras, números y el símbolo de subrayado '_' con una longitud máxima de 64 símbolos. Entonces, las variables válidas son cosas como fish, foo_bar, num1, y las variables inválidas serían 6fish, foo bar o *num.

Ahora, en muchos lenguajes de programación, debe "declarar" una variable antes de poder usarla. Básicamente, esto significa que le dice a la computadora el nombre que desea usar para que pueda asignar un lugar en la memoria para almacenar cualquier valor que decida mantener en esa variable. Con GML, eso no siempre es necesario ya que depende del alcance de la variable. Hay cuatro categorías de variables principales cuando programa con GameMaker Studio 2 y cada una tiene su propio alcance (que puede considerarse como su área de operación o alcance). Estas variables y su alcance se resumen a continuación:

Una variable de instancia se crea dentro de una instancia de un objeto y se considera única para esa instancia, es decir: muchas instancias del mismo objeto pueden tener la misma variable, pero cada variable puede tener un valor diferente, ya que son únicas para cada instancia. Pero, ¿cómo se crea una variable de instancia? Bueno, creas nuevas variables simplemente asignándoles un valor como se muestra en este pequeño ejemplo:

potions = 12;
life = 100;
name = "Jock MacSweeney";
strength = 5.5;
armour = -2;

Como puede ver, solo tiene que dar el nombre y luego un valor (ya sea numérico o una cadena) para establecer esa variable y tenerla lista para su uso dentro de una instancia del objeto que está codificando. Estas variables se pueden usar y modificar de varias maneras desde la instancia, por ejemplo, este código podría estar en un evento de colisión y usarse para quitar una cantidad de la variable "vida":

life -= 5 + armour;

Si "vida" está en 100, ahora tendrá un valor de 97 (100 - (5 + -2) = 97). Ahora, ese es un ejemplo simple, y podría reemplazar "armadura" por el valor real de -2, pero ¿qué sucede si usa ese valor en múltiples lugares y luego decide cambiarlo? ¡Tendría que revisar TODO su código y cambiar cada -2 a cualquiera que sea el nuevo valor, lo cual requiere mucho tiempo y es muy propenso a errores! Pero si usa una variable, todo lo que tiene que hacer es reasignarla a un nuevo valor y el código utilizará automáticamente ese nuevo valor a partir de ese momento, haciendo las cosas mucho más flexibles y mucho más fáciles de solucionar en caso de que haya un problema. También se debe tener en cuenta que incluso si un valor no va a cambiar, es mucho más fácil recordar lo que significa una variable llamada "vida" en lugar de solo mirar un número.

GameMaker Studio 2 tiene una colección de variables de instancia "incorporadas", por lo que debe conocerlas, ya que puede nombrar a una de sus propias variables de instancia de la misma manera o desear tener su propia variable global con el mismo nombre y preguntarse por qué están recibiendo errores. Sin embargo, son fáciles de detectar, ya que se muestran en un color diferente en el editor de código y también aparecen en la barra de autocompletar en la parte inferior.

Hay un par de funciones diseñadas para ayudarlo a manejar las variables de instancia (principalmente para usarlas con Compatibilidad Scrpts para proyectos importados y el código para las acciones de Arrastrar y soltar, pero también se pueden usar en otros lugares):

  • variable_instance_exists
  • variable_instance_get
  • variable_instance_set
  • variable_instance_get_names

Una variable local es aquella que creamos para un evento específico solamente y luego descarta cuando el evento ha finalizado (la única excepción a esto es en los recursos de script, donde un var la variable declarada es local al script y luego se descarta). ¿Por qué los necesitaríamos? Bueno, las variables ocupan espacio en la memoria y puede ser que solo las vamos a usar para una operación o función, en cuyo caso solo necesitamos tenerla en la memoria durante el breve tiempo que se usa. Esto mantiene su base de código limpia y ordenada, además de mantener el espacio de memoria optimizado para las cosas que realmente lo necesitan. Para declarar una variable local usamos la función. var Me gusta esto:

var i, num, str;
i = 0;
num = 24.5;
str = "Hi there!";

Todas las variables creadas anteriormente se "olvidarán" (es decir, se eliminarán de la memoria) al final del evento (o script) en el que se crearon. Debes tener cuidado de que el nombre que das var Las variables declaradas no coinciden con otra variable de instancia dentro del objeto que ejecuta el código, y también se aseguran de que no tenga intención de usar el valor almacenado en esa variable fuera del evento. Declarar. Estas variables se usan mucho en los programas. especialmente en bucles para contar iteraciones, o cuando se usa un valor varias veces en una operación que no se repetirá nuevamente. Aquí hay otro par de ejemplos:

var i = 0;
repeat (10)
   {
   inventory[i] = 0;
   i+=1;
   }

El código anterior crea una variable local llamada "i" y la establece en 0, todo en la misma línea. Tenga en cuenta que en versiones anteriores de GameMaker tenía que declarar sus variables locales primero y luego asignarles valores, pero en esta versión puede declararlas y asignarles un valor al mismo tiempo. El código anterior utiliza esta variable para inicializar una matriz. Como la variable "i" no se utilizará para ninguna otra función en la instancia que no sea ésta, puede tener un alcance local. Aquí hay un ejemplo más:

var xx,yy;
xx = x - 32 +irandom(64);
yy = y - 32 +irandom(64);
instance_create_layer(xx, yy, "Effects", obj_blood);

Aquí hemos utilizado las variables locales para almacenar algunas coordenadas aleatorias que luego usamos para crear una instancia. En este ejemplo, puede ver que no es estrictamente necesario que utilicemos estas variables, sino que lo hacemos por razones de legibilidad y facilidad de uso. Es MUCHO más claro y obvio lo que estamos haciendo allí que si usáramos este código:

instance_create_layer(x - 32 + irandom(64), y - 32 + irandom(64), "Effects", obj_guts);

Otra cosa sobre var las variables declaradas deben tenerse en cuenta... Ya que son exclusivas del evento que las ejecuta, también se pueden usar en cualquier otro caso a través del código. Esto significa que podemos usar estas variables para establecer y cambiar cosas en otros casos utilizando la construcción "con ()" (hay una sección sobre esto en la sección de descripción general de GML del manual). El código real en sí sería algo así:

var num = instance_number(obj_Enemy);
with (obj_Enemy)
   {
   if num>10 instance_destroy();
   }

El código anterior funciona porque el var La variable declarada es local al evento (o script) en el que está contenida, no en la instancia ni en el mundo del juego, por lo que se puede usar en cualquier función en cualquier objeto siempre que esté en el mismo bloque de código.


Una descripción básica de una variable global es aquella que, una vez declarada, no pertenece a ninguna instancia en particular y que todos pueden acceder a ella. Al igual que las variables locales, las variables globales deben declararse, pero a diferencia de una variable local, una variable global permanece en la memoria hasta el final del juego. Por lo tanto, puede crear una variable global para realizar un seguimiento de (por ejemplo) el número de balas que tiene el jugador y luego actualizar esta variable en diferentes puntos del juego. Una variable global no pertenece a ninguna instancia específica y puede ser accedida, cambiada y utilizada por todas las instancias en cualquier momento, pero cualquier cambio realizado en la variable es "global", ya que todas las instancias que usen la variable se verán afectadas por el cambio. Echemos un vistazo a un ejemplo:

global.food = 5;

Declaramos la variable "comida" escribiendo primero "global" y luego una "." para decirle a GameMaker Studio 2 que esta variable es ahora de alcance global. Tendremos que utilizar este método a partir de ahora cada vez que tengamos que acceder o cambiar esta variable de alguna manera. Entonces, hemos creado una nueva variable llamada "comida" y la hemos declarado como global. Ahora, cualquier instancia puede usar y cambiar esta variable de cualquier manera y todas las demás instancias "verán" esto. Por ejemplo, podríamos tener un objeto de comida diferente con el que el jugador colisiona y en el evento de colisión tenemos:

global.food +=1;

También tenemos otro objeto que dibuja este valor de esta manera:

draw_text(32, 32, "food = " + string(global.food));

Con las variables globales podemos cambiar los valores y ver esos cambios reflejados en todas las instancias de los objetos que hacen referencia a esta variable. Al igual que con las variables locales, debe tener cuidado de no nombrar sus variables globales del mismo modo que las variables de instancia, ya que esto le causará problemas y hará que los errores se introduzcan en sus juegos debido a la superposición de variables, lo que puede ser un problema difícil de depurar a veces. En general, debe tener un solo script u objeto que declare todas sus variables globales al comienzo del juego (por ejemplo, en el Evento de inicio de sala), para que estén inicializadas y listas para que las use cualquier instancia, y esto también le brinda un lugar práctico para regresar y referencias en caso de que necesite verificar un nombre de variable.

GameMaker Studio 2 tiene una colección de variables globales "integradas", por lo que debe conocerlas, ya que puede nombrar una de sus variables de instancia iguales o desear tener su propia variable global con el mismo nombre y preguntarse por qué lo hace. obteniendo errores! Sin embargo, son fáciles de detectar, ya que se muestran en un color diferente en el editor de código y también aparecen en la barra de autocompletar en la parte inferior. La mayoría de las variables globales integradas son muy específicas y solo se utilizarán en raras ocasiones y se enumeran en las secciones correspondientes del manual, pero hay una importante que se usa con frecuencia y no se encuentra en ninguna otra parte:

  • async_load

También hay tres variables globales obsoletas integradas que debe conocer (estas variables solo están diseñadas para admitir proyectos heredados de versiones anteriores de GameMaker y no deben usarse en nuevos proyectos ):

  • score
  • health
  • lives

El siguiente método también se puede usar para declarar variables globales, pero solo se incluye para la compatibilidad con versiones anteriores, y no se recomienda que use este método para nuevos proyectos, ya que es posible que las futuras versiones de GameMaker Studio 2 no lo admitan.

El segundo método para crear variables globales es declararlas como tales usando el globalvar declaración, al igual que lo haría con una variable local utilizando la var declaración.

¡IMPORTANTE! los globalvar La declaración está en desuso y solo se admite con fines heredados. Siempre debe usar la global. Identificador para marcar variables globales.


Esta declaración (en desuso) se usaría de la siguiente manera:

globalvar food;
food = 5;

Una vez declarado de esta manera, la variable "alimento" ahora se considera global y no requiere global. prefijo: lo que también significa que es mucho más difícil identificar las variables globales en su código y también es mucho más fácil obtener la superposición de variables, ya que utiliza el mismo nombre de variable en diferentes objetos o desde las extensiones que ha instalado. Una vez declarado de esta manera, se accede a la variable global de la siguiente manera:

food += 2;

o:

draw_text(32, 32, "food = " + string(food));

Como puede ver, sin nada que muestre que la variable es de alcance global, potencialmente se está preparando para que surjan muchos problemas sutiles en su juego, por lo que debe evitarse esta declaración.

Hay un par de funciones diseñadas para ayudarlo cuando se trata de variables globales (principalmente para su uso con scripts de compatibilidad para proyectos importados y el código para las acciones de arrastrar y soltar, pero también se pueden usar en otros lugares):

  • variable_global_exists
  • variable_global_get
  • variable_global_set

Las variables integradas son variables especiales que están "integradas" en los objetos y las salas en el mundo del juego y pueden ser de instancia o de alcance global (pero nunca locales). Algunas variables globales se enumeran en la sección anterior, y las diferentes secciones del manual para sprites, salas, objetos, etc. también describen las variables incorporadas disponibles. Ejemplos de tales variables de instancia incorporadas serían:

  • sprite_index
  • path_scale
  • speed

Y ejemplos de variables globales incorporadas serían:

  • view_xport
  • GM_version
  • room_speed

Las variables incorporadas se pueden cambiar y establecer como la mayoría de las otras variables, y algunas incluso pueden ser matrices, solo que no tiene que configurarlas para crearlas como lo haría con una variable regular, ya que ya se inicializarán a un valor predeterminado.


Si bien no son exactamente variables, las macros son similares a ellas en la forma en que se usan, es decir, son valores con nombre que puede usar a lo largo de su código para reemplazar valores duros. Básicamente, una macro (también llamada constante cuando se usa de esta manera) es una variable con nombre que contiene un valor único constante (o cadena). Puede definir sus propias constantes usando el Editor de secuencias de comandos y luego usarlas en su código y DnD como si fueran variables regulares, con la única diferencia de que no se pueden cambiar en el juego. Por ejemplo, digamos que define la siguiente macro como una constante (tenga en cuenta el "#" anterior y la falta de dos puntos ";" al final):

#macro total_weapons 10

Entonces llamaría a esto en su código así:

if ++pos == total_weapons
   {
   pos = 0;
   }

Tenga en cuenta que no podrá cambiar el valor constante, por lo que un código como este hará que el juego se bloquee:

total_weapons = 11;

Puede definir una macro en cualquier lugar de su código o secuencias de comandos y será precompilada e incluida en su juego como si estuviera allí desde el principio, pero le recomendamos que cree un recurso de secuencia de comandos dedicado y defina todas sus macros allí. ¡Será más fácil organizar y depurar más tarde!

Si necesita que el valor de una macro cambie en tiempo de ejecución, entonces probablemente debería convertirla en una variable global, ya que estos pueden cambiarse de código durante un juego, a menos que establezca que la macro sea una función. Al establecer la macro en una función, significa que esta función se activará cada vez que utilice la macro. Por ejemplo:

#macro col make_colour_hsv(irandom(255), 255, 255)

Entonces llamarías a esta macro algo como esto:

image_blend = col;

El uso de este código hará que la imagen tenga un color diferente cada vez que se use la macro. Vale la pena señalar que también puede dividir macros en varias líneas usando la barra diagonal \ para mostrar dónde se rompe la línea. Un ejemplo sería algo como:

#macro hello show_debug_message("Hello" + \
string(player_name) + \
", how are you today?");

Esto es puramente cosmético, ya que dividir una macro como esta no tendrá ningún efecto sobre el resultado de la macro final, y es simplemente proporcionar soporte para texto de varias líneas en macros que tienen líneas de código más largas.

Una característica muy importante de las macros es que pueden definirse para su uso con Configuraciones específicas (configuraciones), lo que significa que puede tener el mismo nombre de macro pero darle diferentes valores según la configuración seleccionada actualmente. Por ejemplo, si tiene una configuración para anuncios de Android y otra para anuncios de iOS, podría definir una sola macro para mantener el valor de ID de aplicación requerido:

#macro ad_id "";
#macro Android:ad_id "com.yoyogames.googlegame"
#macro iOS:ad_id "com.yoyogames.appstoregame"

Como puede ver, primero asigna el nombre de configuración y luego los dos puntos: y luego el nombre y el valor de la macro. Tenga en cuenta que no puede tener espacios en blanco entre los dos puntos: ni el nombre de la configuración ni el nombre de la macro, de lo contrario obtendrá un error.
