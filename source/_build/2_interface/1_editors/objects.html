El editor de objetos


Los objetos son el recurso que utilizamos para controlar aspectos de un juego y para hacer cosas específicas. La mayoría de las veces tienen un sprite asociado para que los veas en las salas de juego, pero a veces se usan como un controlador "detrás de la escena" para hacer cosas relacionadas con el usuario o para la sincronización, etc. se les puede dar comportamientos y pueden reaccionar a ciertos eventos, así como entre ellos, y la mayoría de las cosas que se ven en un juego se basan en los objetos y sus interacciones. Ahora, decimos "basado en" porque en realidad no coloca objetos directamente en las salas de juego, sino que coloca instancias de estos objetos que son básicamente copias (o clones, si lo prefiere) del recurso de objeto. Esto es algo muy importante para recordar, ya que las instancias y los objetos no son lo mismo y cada uno tiene su propio conjunto de funciones que pueden afectarlos.

Para aclarar esto, piense en un juego hecho con GameMaker Studio 2. Todos los personajes, monstruos, bolas, paredes, etc. son todos los objetos que creas en el árbol de recursos. A continuación, arrastre el objeto a una habitación en el Editor de salas para crear una instancia copiada de la plantilla de objetos básica. La instancia en la sala se puede cambiar en el editor mismo o mediante el código, de modo que se pueda escalar, colorear o modificar otros detalles desde el objeto "base". Entonces, cuando hablamos de algo que afecta o cambia una instancia, queremos decir que afecta a esa copia particular de un objeto en una habitación. Solo esa única instancia está siendo afectada mientras que el resto no lo está. Pero, cuando hablamos de afectar o cambiar un objeto, queremos decir que estamos modificando el objeto en el árbol de recursos y todo lo que hagamos al mismo se reflejará en todas las instancias creadas desde ese momento también. Entonces, el objeto es la plantilla para la instancia, y la instancia es lo que colocamos en una habitación para hacer nuestro juego.

Para crear un objeto en su juego, puede hacer clic derecho en la carpeta de recursos de objeto y haga clic en la opción Crear. Esto abrirá la ventana que se muestra en la parte superior de esta página con las siguientes secciones:

Aquí puedes darle un nombre a tu objeto. Este nombre nunca debe contener espacios ni caracteres no autorizados (solo se permiten los caracteres alfanuméricos y la barra inferior "_") y debe ser corto pero memorable para que pueda identificar los recursos más adelante cuando tenga muchos de ellos. Por ejemplo, muchas personas usan un prefijo o un sufijo para identificar las diferencias entre los recursos, como "obj_Ship" o "Ship_obj".

A continuación, puede elegir el sprite para el objeto si desea que tenga uno. Haga clic con el botón izquierdo del ratón. en el cuadro de sprites y el Explorador de activos aparecerá con una lista de todos los sprites disponibles del árbol de recursos:

A continuación, puede seleccionar el que desea utilizar para el objeto. Con un sprite asignado al objeto, puede hacer clic en el botón Editar Sprite para cambiar las propiedades del sprite, o si no tiene un sprite para el objeto pero desea crear uno, puede hacer clic en el botón Agregar Sprite para crear un nuevo recurso sprite y asignarlo. También puede editar la imagen del recurso sprite que se ha asignado al hacer clic en el botón Editar imagen para abrir el editor de imágenes.


Debajo del área de asignación de sprites, tiene un botón para elegir una Máscara de colisión específica. Por defecto, un objeto utilizará las propiedades de colisión del sprite asignado, pero hay casos en los que es posible que desee que las colisiones se basen en una forma de máscara diferente. Por ejemplo, puede tener un juego de arriba hacia abajo donde el sprite principal tiene una máscara de colisión rectangular, pero para hacer las colisiones más suaves, una máscara circular puede ser mejor. En estos casos, puede hacer clic en el botón Máscara de colisión y seleccionar otro recurso de sprite, por lo que ahora las instancias del objeto dibujarán el sprite asignado, pero basarán las colisiones en el sprite de la máscara.

Visible indica si las instancias de este objeto son visibles cuando se inicia la sala. Normalmente, la mayoría de los casos son visibles, pero a veces es útil tener invisibles; por ejemplo, puede usarlos como puntos de referencia para controlar un monstruo en movimiento, o para mantener un seguimiento de los valores y realizar ciertas acciones cronometradas. Los objetos invisibles seguirán reaccionando a los eventos y si tienen un sprite o una máscara asignados a ellos, también reaccionarán si otras instancias chocan con ellos... simplemente no se los puede ver y no realizan su evento de dibujo. Por defecto, la bandera visible está marcada.


Cuando marca una instancia como sólida, le está diciendo a GameMaker Studio 2 que debe intentar resolver cualquier colisión antes de desencadenar un evento de colisión. Esta es una funcionalidad muy básica, y en realidad todo lo que hace es mover la instancia de nuevo a la posición que tenía en el momento antes de que se detectara una colisión antes de realizar cualquier código o DnD™ que esté en el evento en sí, y como tal es de uso limitado.


Persistente: un objeto persistente es aquel que no desaparece cuando se cambia la habitación, sino que "persiste" y se traslada a la nueva habitación. Solo desaparecerá cuando lo destruyas explícitamente, ya sea con una acción de destruir o en código. Esto significa que si coloca un objeto persistente en una sala, seguirá estando disponible en todas las salas subsiguientes y continuará realizando las acciones asignadas a los eventos en sus propiedades de objeto, excepto el evento create, que solo se activa una vez cuando Se crea inicialmente y no cuando cambias a una nueva sala.

Esto es útil para cuando tiene, por ejemplo, un personaje principal que se mueve de una habitación a otra y desea mantener las variables dentro de ese objeto de la misma manera, pero debe tener mucho cuidado de crear y destruir objetos persistentes correctamente, ya que puede hacerlo fácilmente. consigue errores arrastrándose en tu proyecto por error.

Tenga en cuenta también que un objeto persistente seguirá teniendo activados los eventos Game Start, Game End, Room Start y Room End, sin embargo, si reinicia el juego (con, por ejemplo, la función game_restart ()), todos los objetos persistentes se eliminarán y solo existe cuando se crea de nuevo por el juego. También tenga en cuenta que si desactiva un objeto persistente, ya no pasará de una habitación a otra a menos que se vuelva a activar antes de que se active el evento Fin de habitación.


Cuando marca la casilla Física de usos, le está diciendo a GameMaker Studio 2 que este objeto debería ser parte de un mundo de física. Al marcar esto también se abrirá una ventana encadenada adicional donde puede definir las propiedades físicas de todas las instancias de este objeto. Más información sobre esto se puede encontrar más abajo en esta página en la sección de Física.

La sección de opciones define las diferentes opciones de propiedades para su objeto y cómo se comportarán las instancias cuando se coloquen en una habitación. Aquí agrega eventos, que son los "bloques de construcción" de su código de juego, establece si el objeto es tener un padre y también establece las propiedades físicas si desea habilitar las funciones físicas.

Entonces, ¿qué son los eventos? Básicamente, estos son momentos discretos en el bucle del juego donde las cosas están hechas para suceder según lo que usted haya programado para ellos. GameMaker Studio 2 funciona con ciclos de estos eventos: desde el momento en que se inicia una sala hasta el momento en que finaliza, se ejecuta un bucle de juego donde cada paso (un paso es un momento en el tiempo del juego, regido por la configuración de la velocidad de la sala) a se ejecutan o verifican series de eventos, y puede elegir colocar código o acciones DnD en sus objetos que respondan a estos eventos.

Cuando trabaje con objetos en el IDE de GameMaker Studio 2, puede configurar las jerarquías padre / hijo. Esto se hace haciendo clic en el botón "Padre" en el editor de objetos y luego seleccionando otro objeto del árbol de recursos para que sea el "padre" del que se está editando:

Entonces, cada objeto en tu proyecto de juego puede tener un objeto padre, pero ¿qué significa esto? Bueno, cuando un objeto tiene un padre asignado, puede compartir código, acciones y eventos con ese padre. Este intercambio se denomina "herencia" y un objeto que tiene un padre se llama un objeto "hijo". Un hijo no solo puede compartir el código con su padre, sino que también puede hacer verificaciones y ejecutar código en los objetos padres y también incluirá automáticamente los objetos hijos, lo que ahorra mucho tiempo y energía.

Si eso suena complicado, entonces otra forma de ver un objeto padre es como una forma de "agrupar" objetos bajo el mismo paraguas y hacer que compartan ciertas cosas sin perder su propia identidad única. Tal vez eso todavía no aclara demasiado las cosas, así que vamos a dar algunos ejemplos...

Digamos que tienes un objeto "jugador" y cuatro objetos "enemigos" diferentes. Ahora, quieres que el jugador muera si toca cualquiera de estos cuatro objetos y esto normalmente implicaría cuatro eventos de colisión diferentes con cuatro conjuntos diferentes de acciones o códigos, uno para cada uno de los objetos enemigos. PERO si hacemos un objeto principal para todos los enemigos, entonces podemos crear un evento de colisión solo con el objeto principal y se disparará sin importar cuál de los cuatro objetos enemigos "menores" toque al jugador. ¡Cosas prácticas! En el objeto real de GameMaker Studio 2 algo como esto:

A la izquierda tenemos cuatro eventos de colisión separados y a la derecha tenemos un evento de colisión único, ya que hemos creado un objeto "padre" y le hemos asignado todos los objetos enemigos. Tenga en cuenta que un objeto principal no tiene que tener ningún evento o código en él...

Otro ejemplo de crianza sería si tiene un juego en el que desea crear 10 objetos de aspecto diferente y hacer que todos se comporten de la misma manera. Para eso, crearía un objeto principal y tendría todas sus acciones de comportamiento o código en los eventos requeridos, y luego crearía sus diez objetos sin acciones o código, pero con diferentes sprites, y les asignaría su objeto principal. Ahora, cuando coloca estas instancias en una habitación, todas se comportarán igual, pero se verán diferentes, ya que "heredarán" los eventos del padre.

Finalmente, puede usar la crianza para "mezclar y combinar" eventos y comportamientos. Ilustraremos esto con un ejemplo final... Digamos que quieres dos monstruos: uno que se mueve hacia arriba y hacia abajo, mientras que otro que se mueve hacia la izquierda y hacia la derecha, pero también quieres que los dos tengan la misma salud, dispara a la Jugador y herir al jugador si chocan con ellos. En este caso, puede ver que casi todos los eventos deben tener las mismas acciones, excepto una o dos que gobiernan el movimiento. Entonces, nuevamente, podemos hacer que un objeto sea el padre del otro, pero en este caso también definimos ciertos eventos para el objeto hijo. Estos eventos "anulan" los eventos principales, lo que significa que cada vez que un evento para el objeto secundario contiene acciones, éstas se ejecutan en lugar de las acciones contenidas en el evento del elemento primario. Si también desea ejecutar el evento principal, puede llamar al evento denominado "heredado" utilizando la función event_inherited(), o la acción DnD™ Call Parent Event.

Si elige Heredar, el editor de código se abrirá con la función event_inherited() ya se agregó a él (o la acción Llamar al evento principal si se usa DnD™). Si elige Anular el evento, la ventana de código también se abrirá, solo que ahora event_inherited() La función no se llamará, por lo que cualquier cosa que agregue aquí se ejecutará en lugar del código en el objeto principal.

Siempre que apunte a un objeto principal en el código, el código también se aplicará a los "hijos" del objeto principal también. Esto sucede cuando, en una acción, indica que la acción debe aplicarse a instancias de un determinado objeto, y en el código sucede cuando usa la with() declaración. Funcionará así también cuando llames a funciones de código, como instance_position, instance_number, etc. donde, si proporciona un objeto principal, todas las instancias de las instancias principal y secundaria se incluirán en las comprobaciones. Finalmente, la crianza de los hijos funciona cuando también te refieres a variables en otros objetos, como en el ejemplo del monstruo anterior, si configuro la velocidad del enemigo 1 en 10, entonces la velocidad del enemigo 2 también llegará a diez, ya que es un objeto secundario del enemigo 1.

En general, se considera una buena práctica en la mayoría de los casos crear un objeto primario base y hacer que este objeto base contenga todo el comportamiento predeterminado, pero nunca use una instancia de él en el juego. En su lugar, use todos los objetos secundarios y solo use el padre en situaciones como las que he descrito anteriormente, para colisiones, para hacer referencia a variables, etc. ¡También debe darse cuenta de que los padres también pueden tener padres! Obviamente, no puede crear un ciclo de "padre 1 es hijo de padre 2 es hijo de padre 1" pero puede crear lo que se llama "jerarquía de objetos" donde "parent3 es hijo de padre2 es hijo de parent1". Esto es extremadamente útil para mantener su juego estructurado y le recomendamos encarecidamente que aprenda a usar este mecanismo.

Una vez que haya configurado su forma, deberá definir las propiedades físicas básicas que debe tener su objeto. Estos se configuran cambiando los siguientes parámetros:

La densidad de algo se define como su masa por unidad de volumen, lo que básicamente significa la cantidad de masa acumulada en el espacio que ocupa en el mundo. Por lo tanto, un globo tendría una densidad baja, ya que tiene poca masa que ocupa un gran espacio, mientras que una barra de plomo tendría una densidad alta, ya que tiene una gran masa, pero ocupa poco espacio. En el mundo de la física 2D de GameMaker Studio 2, la masa se calcula automáticamente a partir de los valores que ingresa para la densidad y por el área de superficie de la forma que define para el accesorio. Esto tendrá un impacto directo en la cantidad de inercia que tiene un artefacto y en cómo reacciona a las colisiones y fuerzas, por lo que si crea una forma pequeña con una densidad alta, tendrá una masa muy grande (como una barra de plomo). pero si define una forma grande con una densidad baja, tendrá una masa mucho más pequeña (como un globo).


En física, la restitución se define como " el retorno de un objeto o sistema a su estado original después de la deformación elástica ", pero como los accesorios en el GameMaker Studio 2 son cuerpos realmente rígidos y no pueden deformarse, la restitución es realmente una forma de decir cómo "hinchable" es el accesorio. Esta configuración afectará a cuánto "rebota" un objeto cuando choca con otros objetos y es dependiente de otras fuerzas que actúan en la instancia como la gravedad y la fricción.


De forma predeterminada, el grupo de colisión para todos los objetos físicos es 0, y esto significa que interactuarán como es normal, lo que a su vez significa que deben tener un evento de colisión para interactuar (un simple comentario hará para desencadenar la colisión) de lo contrario, no habrá colisión. estar registrado Sin embargo, también puede especificar que un objeto debe pertenecer a un grupo de colisión específico. Al asignar objetos a un grupo de colisión positivo (es decir, tres objetos están todos en el grupo " 2 "), le está diciendo a GameMaker Studio 2 que las instancias de estos objetos siempre deben colisionar, incluso si no tienen nada en su evento de colisión, obedeciendo las propiedades físicas que les ha asignado y la sala en la que están. A la inversa, si tiene objetos con un grupo de colisión negativo (es decir, cuatro objetos asignados " -1 "), le está diciendo a GameMaker Studio 2 que estos nunca deben colisionar, y cualquier colisión Los eventos entre ellos serán ignorados.

NOTA: el uso de grupos de colisión aumenta dramáticamente el procesamiento requerido por el sistema de física y debe usarse solo cuando sea absolutamente necesario, y debe tener la menor cantidad de grupos posible.

La amortiguación se utiliza para reducir la velocidad mundial física de una instancia, y difiere de la fricción en que la fricción solo se produce cuando dos instancias con formas de colisión (dispositivos) están en contacto. La amortiguación también es mucho más barata de simular que la fricción, pero tenga en cuenta que la amortiguación no es un reemplazo para la fricción; Los dos efectos pueden y deben ser usados ​​juntos.


Si piensa en algún objeto giratorio en el "mundo real", a menos que tenga un motor o esté en el espacio, se ralentiza con el tiempo debido a la influencia de fuerzas externas (como la fricción con el aire que lo rodea). Podemos usar esta opción para simular este efecto y reducir la velocidad de rotación de las instancias en el mundo de la física, ya que, sin ella, cualquier instancia de rotación continuaría girando infinitamente.


La fricción es la fuerza que resiste el movimiento relativo de los elementos materiales que se deslizan entre sí, lo que en el mundo físico de GameMaker Studio 2 traduce como la pérdida de impulso causada por la colisión de dos instancias con accesorios vinculados a ellos. Entonces, cuando dos instancias chocan, su movimiento se ve afectado por este valor, con una alta fricción que causa una mayor pérdida de impulso que un valor más bajo.

Los diferentes tipos de variables que puede crear se enumeran a continuación:

  • Real: un número real es cualquier número que no sea un entero entero y puede ser positivo o negativo. Entonces, 124.5, 45639.566546456, 0.9, -45.5, etc... son todos ejemplos de números reales. Todos los números reales se almacenan como valores de punto flotante de 64 bits (dobles). Tenga en cuenta que puede establecer un rango de valores para la salida real desde las Opciones de Variable (ver más abajo para más detalles).

  • Entero: un entero es un número entero y puede ser positivo o negativo, por ejemplo 30004, 19, 0, -300. Tenga en cuenta que puede establecer un rango de valores para el entero de salida desde las Opciones de variables (consulte a continuación para obtener más detalles).

  • Cadena: una cadena es cualquier cosa que se haya colocado entre comillas dobles, por ejemplo, "pez", "Hola mundo" o "12345".

  • Booleano: un booleano es un valor que puede ser true o false. En la ventana Variables del Editor de objetos, esto se muestra simplemente como un cuadro que usted verifica true y desmarque false.

  • Expresión: una expresión es una frase matemática que puede contener números ordinarios, variables, cadenas o funciones GML, así como uno o más operadores. Por ejemplo sqrt(85 * 6) + 5.5 es una expresion.

  • Recurso: un recurso es simplemente cualquiera de los recursos que ha definido en el árbol de recursos. Cuando seleccione este tipo de recurso, puede hacer clic en el Explorador de activos abiertos para abrir el Explorador de activos y seleccionar el recurso que necesita. Tenga en cuenta que puede configurar filtros para los activos que se muestran en el Explorador de activos desde las Opciones de variables (consulte a continuación para obtener más detalles).

  • Lista: Seleccionar un tipo de lista de entrada significa que luego puede crear una selección de valores (que pueden ser cadenas, reales, expresiones, etc.) de los cuales puede elegir uno o más para que la variable regrese. Para definir los elementos que van en la lista, primero debe agregarlos a las Opciones de variables (consulte a continuación para obtener más detalles) y luego podrá elegir el elemento como el valor predeterminado para la variable en la ventana principal desde el menú desplegable. menú desplegable: Tenga en cuenta que si tiene Marcado múltiple en las Opciones de lista (ver a continuación), la variable se convertirá en una matriz de todas las opciones seleccionadas.

  • Color: El tipo de color es para que usted defina un valor de color que se almacenará en la variable. Puede ingresar un número real (de 0 a 16777216, tendrá un valor alfa de 255), o un valor hexadecimal (en el formato de $ RRGGBBAA), o puede hacer doble clic en la muestra de color para abrir el Selector de color y definir el color allí.


Algunos de los diferentes tipos de variables le darán opciones para definir el valor de salida, y puede abrir estas opciones haciendo clic en sobre el botón:

  • Opciones reales / enteras: Los tipos de datos reales y enteros tienen una opción para establecer un rango de valores. Cuando marque esto, puede ingresar un valor de inicio y un valor final y, en lugar de mostrar un valor fijo para la variable en la ventana de Variables del objeto, tendrá un control deslizante que está sujeto a estos valores:

  • Opciones de recursos: Las opciones de recursos se abrirán con una lista de todos los tipos de recursos disponibles y, de forma predeterminada, se seleccionarán todos. Sin embargo, puede hacer clic en cualquiera de ellos para seleccionar solo ese tipo, o puede usar / + o + para seleccionar múltiples tipos de recursos. Una vez que haya configurado esto, solo el tipo de recurso seleccionado será visible en el explorador de activos para la variable.

  • Opciones de lista: Aquí puede agregar los diferentes valores que desea incluir como posibles valores de retorno para la variable de tipo de lista. Simplemente haga clic en el Botón Agregar elemento de lista para agregar el nuevo valor (que puede ser una cadena, real, entero, nombre del recurso) y luego en la ventana de variables del editor de objetos aparecerá en la lista desplegable para el valor predeterminado y puede seleccionar el que sea necesario: Si ha marcado la opción Selección múltiple, entonces la lista desplegable tendrá casillas de verificación junto a cada uno de los valores que desea que devuelva la variable de lista (y la variable se convertirá en una matriz para contener cada uno de ellos):
